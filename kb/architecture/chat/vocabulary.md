# Chat Architecture Vocabulary

Terms covering the chat pipeline, spawn lifecycle state machine, event system, normalization, command handling, and recovery. For the full vocabulary index see [../../vocabulary.md](../../vocabulary.md).

| Term | Definition | See also |
|---|---|---|
| **Backend acquisition** | The interface (`BackendAcquisition`) that creates a harness connection, wires a normalizer and observer, and returns a live backend handle to the chat pipeline. `ColdSpawnAcquisition` is the concrete fresh-start implementation. | [backend-acquisition.md](backend-acquisition.md) |
| **Backend handle** | A per-execution harness connection handle held by `ChatSessionService`; allows prompts, cancels, and approval/input responses to reach the running harness. | [runtime-and-sessions.md](runtime-and-sessions.md) |
| **Background spawn** | A spawn created with `--bg`, where `meridian spawn` returns immediately with the spawn ID rather than blocking. The harness process runs asynchronously; callers use `meridian spawn wait` to block until completion. | [../../concepts/spawn-lifecycle.md](../../concepts/spawn-lifecycle.md) |
| **Chat command** | A frontend-to-backend command envelope with fields `type`, `command_id`, `chat_id`, `timestamp`, `payload`. Supported types: `prompt`, `cancel`, `approve`, `answer_input`, `close`, `revert`, `swap_model`, `swap_effort`. | [command-system.md](command-system.md) |
| **Chat event** | The normalized event envelope produced by the chat backend pipeline. Fields: `type`, `seq`, `chat_id`, `execution_id`, `timestamp`, optional `turn_id`, `item_id`, `request_id`, `payload`, `harness_id`. | [event-pipeline.md](event-pipeline.md) |
| **Chat event index** | A SQLite projection over the chat event log (`ChatEventIndex`). Tables: `events`, `files`, `turns`, `checkpoints`, `token_usage`, `spawns`. Rebuilt from the JSONL source of truth on recovery. | [event-pipeline.md](event-pipeline.md) |
| **Chat event log** | The append-only JSONL file (`ChatEventLog`) that is the source of truth for a chat session. Assigns monotonic per-chat sequence numbers; reads tolerate truncated tails. | [event-pipeline.md](event-pipeline.md) |
| **Chat event observer** | The bridge (`ChatEventObserver`) between one harness execution and the chat pipeline. Normalizes raw `HarnessEvent`s, filters by `execution_id`, and injects `execution_generation` into payloads. | [backend-acquisition.md](backend-acquisition.md) |
| **Chat event pipeline** | The persistence-first event pipeline (`ChatEventPipeline`). Sequence: validate ownership → queue → append to JSONL log → upsert SQLite index → broadcast to WebSockets → invoke turn-completed callback. | [event-pipeline.md](event-pipeline.md) |
| **Chat ID** | A session identifier with the prefix `c` (e.g., `c42`). Monotonically assigned from a counter in the runtime root. The top-level primary session's chat ID is propagated to all descendant spawns via `$MERIDIAN_CHAT_ID`. | [../../concepts/spawn-lifecycle.md](../../concepts/spawn-lifecycle.md) |
| **Chat runtime** | The top-level registry (`ChatRuntime`) for live and recovered chats. Owns live entries, persisted-only records, and recovery state. Starts by recovering from disk once, then starts live pipelines. | [runtime-and-sessions.md](runtime-and-sessions.md) |
| **Chat session service** | The per-chat session state machine (`ChatSessionService`). States: `idle → active → draining → closed`. Owns backend reacquisition when a prior handle dies. | [runtime-and-sessions.md](runtime-and-sessions.md) |
| **Chat state** | The lifecycle values of a `ChatSessionService` instance: `idle`, `active`, `draining`, `closed`. | [runtime-and-sessions.md](runtime-and-sessions.md) |
| **Cold spawn acquisition** | The concrete `BackendAcquisition` strategy that starts a fresh harness execution, registers the observer before starting the spawn (critical ordering invariant), and always uses `PersistentDrainPolicy` for chat spawns. | [backend-acquisition.md](backend-acquisition.md) |
| **Command result** | The backend synchronous ack from command dispatch: `accepted` or `rejected`, with optional error string. Distinct from `CommandAck` (frontend) — this is the backend type in `lib/chat/commands.py`. | [command-system.md](command-system.md) |
| **Control socket** | A per-spawn injection endpoint. POSIX: a Unix domain socket. Windows: a localhost TCP listener with a `.port` discovery file. Currently supports `user_message` injection. | [backend-acquisition.md](backend-acquisition.md) |
| **Depth** | The spawn nesting level, carried in `$MERIDIAN_DEPTH`. `0` or absent means root (the primary interactive session). `1` is a spawn launched by the primary. The reaper uses depth to restrict root-only side effects. | [../../concepts/spawn-lifecycle.md](../../concepts/spawn-lifecycle.md) |
| **Drain loop** | The streaming event processing loop inside `SpawnManager` that iterates raw `HarnessEvent`s from the connection, persists them to `history.jsonl`, dispatches observers, and classifies terminal events via the drain policy. | [../state-system.md](../state-system.md) |
| **Drain policy** | The streaming policy controlling when to stop consuming harness events. Two implementations: `SingleTurnDrainPolicy` (first terminal event ends the spawn) and `PersistentDrainPolicy` (succeeded turns emit a turn boundary and continue). | [backend-acquisition.md](backend-acquisition.md) |
| **Event family** | Top-level namespace grouping chat event types. Families: `chat`, `work`, `turn`, `content`, `item`, `files`, `spawn`, `request`, `checkpoint`, `model`, `runtime`, `extension`. | [event-pipeline.md](event-pipeline.md) |
| **Event normalizer** | A stateful per-execution component (`EventNormalizer`) that maps raw `HarnessEvent`s to `ChatEvent`s. Stateful because tool-use block input accumulates across deltas. Per-harness implementations: claude, codex, opencode. | [normalization.md](normalization.md) |
| **Event observer** | The streaming post-persist observer protocol (`EventObserver`). Implementations: `CallbackObserver`, `QueuedObserver`. Managed by `EventObserverRegistry` per spawn. | [../state-system.md](../state-system.md) |
| **Event observer registry** | Per-spawn observer management (`EventObserverRegistry`). Supports register, unregister, dispatch, complete, and shutdown. Delivery is asynchronous; queue overflow emits a warning instead of blocking the drain loop. | [../state-system.md](../state-system.md) |
| **Execution generation** | A monotonic counter embedded in chat events to fence stale events from a prior harness execution. `ChatEventObserver` injects `execution_generation` into payloads; `on_turn_completed()` and `on_execution_died()` use it to ignore stale callbacks. | [runtime-and-sessions.md](runtime-and-sessions.md) |
| **Finalizing** | The transient spawn status entered via `mark_finalizing` CAS after the harness process exits. Indicates the runner is finishing drain, report extraction, and state persistence. Treat as active when polling. Becomes `succeeded` or `failed` once finalization completes. | [../../concepts/spawn-lifecycle.md](../../concepts/spawn-lifecycle.md) |
| **Foreground spawn** | A spawn created without `--bg`, where `meridian spawn` blocks until the harness process completes and returns the outcome in the terminal. | [../../concepts/spawn-lifecycle.md](../../concepts/spawn-lifecycle.md) |
| **HITL** | Human-in-the-loop. A point during harness execution where the agent sends an approval or input request requiring a user response before continuing. Surfaced in the chat UI as an approval card or user-input prompt. | [command-system.md](command-system.md) |
| **Lifecycle event** | The immutable shared event envelope (`LifecycleEvent`) for spawn lifecycle transitions. Carries spawn context, metrics, and a deterministic UUIDv5 ID keyed by subject + event type + sequence. Used by hooks, telemetry, and debug observers. | [../state-system.md](../state-system.md) |
| **Mark finalizing** | A compare-and-swap (CAS) operation that transitions a spawn's status from `running` to `finalizing` immediately after the harness process exits, before drain and report extraction begin. Enables the reaper to distinguish `orphan_finalization` from `orphan_run`. | [../../concepts/spawn-lifecycle.md](../../concepts/spawn-lifecycle.md) |
| **Orphan finalization** | A spawn classified as failed by the reaper because it was in `finalizing` status with no active runner (heartbeat stale, runner process gone). Indicates a crash during the finalization window. Distinguished from `orphan_run` by the pre-crash status transition. | [../../concepts/spawn-lifecycle.md](../../concepts/spawn-lifecycle.md) |
| **Orphan run** | A spawn classified as failed by the reaper because it was in `running` status with no active runner (heartbeat stale, runner process gone). Indicates a crash during active execution. The most common failure mode in practice. | [../../concepts/spawn-lifecycle.md](../../concepts/spawn-lifecycle.md) |
| **Persistent drain policy** | The drain policy for chat-backed spawns (`PersistentDrainPolicy`). Succeeded turns emit a synthetic turn boundary event and continue draining. Failures and cancellations terminate the drain. | [backend-acquisition.md](backend-acquisition.md) |
| **Replay service** | The chat pipeline service (`ReplayService`) that replays persisted event history to newly connected WebSocket clients, then tails the live fanout queue. Registers before replay to avoid gaps. | [event-pipeline.md](event-pipeline.md) |
| **Report** | A markdown summary extracted from harness output at finalization, persisted as `spawns/<id>/report.md`. Auto-extraction attempts structured harness output first, then falls back to a text heuristic. | [../../architecture/launch-system.md](../../architecture/launch-system.md) |
| **Single turn drain policy** | The drain policy for non-chat spawns (`SingleTurnDrainPolicy`). The first terminal event from the harness ends the spawn and closes the drain loop. | [../../concepts/spawn-lifecycle.md](../../concepts/spawn-lifecycle.md) |
| **Spawn application service** | The orchestration shell (`SpawnApplicationService`) around the core lifecycle, state, and cancel primitives. Owns `prepare_spawn()` (builds launch context before row creation), `cancel()`, `complete_spawn()`, `archive()`, and `update_metadata()`. | [../state-system.md](../state-system.md) |
| **Spawn context ref** | A typed context reference (`SpawnContextRef`, `SessionContextRef`, `ContextRef`) for continuation prompts. Resolves to prompt content from a prior spawn or session. Used by `spawn continue` to chain work. | [../../concepts/spawn-lifecycle.md](../../concepts/spawn-lifecycle.md) |
| **Spawn failure** | The sentinel payload (`SpawnFailure`) written alongside failed spawns. Used by crash recovery and reconciliation paths to distinguish a clean failure from an orphaned spawn. | [../../concepts/state-model.md](../../concepts/state-model.md) |
| **Spawn ID** | A unique identifier with the prefix `p` (e.g., `p42`). Monotonically assigned. Used to reference a spawn in CLI commands, API calls, and artifact paths. | [../../concepts/spawn-lifecycle.md](../../concepts/spawn-lifecycle.md) |
| **Spawn lifecycle service** | The authoritative seam (`SpawnLifecycleService`) for spawn state transitions. Operations: `start`, `mark_running`, `record_exited`, `finalize`, `mark_finalizing`, `cancel`, `update_metadata`. Hooks are dispatched after store writes and never block the transition. | [../state-system.md](../state-system.md) |
| **Spawn manager** | The streaming spawn runtime registry (`SpawnManager`). Owns per-spawn: active harness connections, drain tasks, completion futures, control sockets, heartbeat tasks, and post-persist observers. Used by `ColdSpawnAcquisition` for chat-backed spawns. | [backend-acquisition.md](backend-acquisition.md) |
| **Spawn record** | The materialized spawn state (`SpawnRecord`) produced by replaying spawn JSONL events through `reduce_events()`. The authoritative in-memory representation of a spawn's current state. | [../state-system.md](../state-system.md) |
| **Spawn request** | The `SpawnRequest` frozen Pydantic DTO representing caller intent at launch time: prompt, model, harness, skills, session flags, budget, etc. JSON-safe, no derived state. Input to `build_launch_context()`. | [../../architecture/launch-system.md](../../architecture/launch-system.md) |
| **Spawn session** | The per-spawn container within `SpawnManager` holding the active harness connection handle, drain task, observers, and completion state. Created by `SpawnManager.start_spawn()`. | [backend-acquisition.md](backend-acquisition.md) |
| **Spawn status** | The current lifecycle state of a spawn. Active: `queued`, `running`, `finalizing`. Terminal: `succeeded`, `failed`, `cancelled`. The reaper transitions stale active statuses to `failed` with a reason code. | [../../concepts/spawn-lifecycle.md](../../concepts/spawn-lifecycle.md) |
| **Turn boundary** | A synthetic event type (`meridian/turn_completed`) emitted by `PersistentDrainPolicy` at the end of each successful harness turn. Signals to the chat pipeline that a turn has completed without ending the drain session. | [backend-acquisition.md](backend-acquisition.md) |
